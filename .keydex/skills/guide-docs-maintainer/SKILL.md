维护 `admin_frontend/public/guide-docs/md/` 目录下的 MD 说明文档。MD 文件是唯一事实来源，浏览器通过 `index.html` Shell 自动渲染。

## 模式判断

首先判断用户任务中是否**提供了 MD 文档路径**：

| 条件 | 模式 | 执行流程 |
|------|------|----------|
| 用户提供了 MD 文件路径 | **模式A** | 直接执行【MD 写入流程】 |
| 用户未提供 MD 文件，或说"创建"、"编写" | **模式B** | 执行【MD 创作流程】 |

---

## 模式A：MD 写入流程（用户提供 MD 时）

### Step 1: 读取 MD 文档

读取用户指定的 MD 文件，理解内容和结构。

### Step 2: 了解现有章节结构

读取 `admin_frontend/public/guide-docs/nav.json`，了解：
- 现有章节及其 `file` 名称
- 五大块分类结构

### Step 3: 评估策略

根据 MD 内容，判断：
1. **更新现有章节**：`nav.json` 中已有对应 `file`，直接覆盖对应 `md/{file}.md`
2. **新增章节**：主题全新，确定归属的五大块分类，分配 `file` 名称，写入新 MD 文件并更新 `nav.json`

给出更新清单：

```
## 更新清单

| 操作 | 目标文件 | 说明 |
|------|----------|------|
| 更新 | guide-docs/md/xxx.md | 章节内容 |
| 更新（如新增章节） | guide-docs/nav.json | 添加导航条目 |
```

等待用户确认后执行。

### Step 4: 写入文件（用户确认后）

- 写入/覆盖 `admin_frontend/public/guide-docs/md/{file}.md`
- 若新增章节：更新 `admin_frontend/public/guide-docs/nav.json`，在正确位置插入条目

### Step 5: 输出结果

列出写入/更新的文件路径列表。

## 模式B：MD 创作流程（无文档时）

### Step 1: 了解现有章节结构

读取 `admin_frontend/public/guide-docs/nav.json` 和 `admin_frontend/public/guide-docs/md/` 目录，了解：
- 现有章节内容范围
- 文档风格和结构

### Step 2: 信息收集

**智能跳过机制**：先分析用户任务描述中已包含的信息：

| 信息项 | 判断标准 | 示例 |
|--------|----------|------|
| 文档主题 | 是否明确提到功能/模块名称 | "记忆策略功能"、"Agent配置" |
| 目标读者 | 是否提到读者类型 | "实施人员"、"运营同事" |
| 文档类型 | 属于五大块哪一类 | "操作说明"、"最佳实践" |
| 涉及功能 | 是否提到具体功能点 | "记忆配置"、"应用管理" |

**跳过规则**：如果任务描述已包含某信息项，不要重复询问。只询问缺失或不明确的信息项。

**收集问题（最多6问）**：

1. 文档主题是哪个功能/模块？
2. 目标读者是谁？（实施/运营/开发/用户）
3. 属于五大块的哪一类？（基础原理/最佳实践/对外接口/版本更新/操作说明）
4. 需要涵盖哪些功能点或操作步骤？
5. 是否有参考的现有文档或界面？
6. 文档的优先级或紧急程度？

### Step 3: 生成文档大纲

基于收集的信息，生成说明文档大纲：

```markdown
# [文档标题]

## 文档信息
- **主题**：[功能/模块名称]
- **目标读者**：[读者类型]
- **文档类型**：[五大块分类]
- **预计长度**：[短/中/长]

## 内容大纲

### 一、简述
### 二、配置/使用顺序
### 三、进入/访问方式
### 四、详细说明
### 五、编辑/更新/删除（如适用）
### 六、最佳实践（如适用）
### 七、常见问题（FAQ）
### 八、附录（术语、菜单路径等）
```

### Step 4: 用户确认

向用户展示大纲，确认：
- 内容结构是否符合预期
- 是否需要调整章节顺序
- 是否有遗漏的功能点

### Step 5: 生成 MD 文档

根据大纲和用户需求，生成完整的 MD 说明文档。

**对外说明文档写作规范**：

内容面向**实施人员、运营同事**等最终用户，**禁止**出现：
- 代码实现细节
- 仓库文件路径
- "与当前实现对齐"、"按项目宪法"等内部术语
- 技术架构描述
- 内部注释或 TODO

**应该包含**：
- **功能简述**：一句话说明这是什么功能
- **使用场景**：什么时候用、给谁用
- **操作步骤**：界面点击路径、字段填写说明
- **界面字段表格**：字段名、是否必填、怎么填
- **常见问题**：实施/运营可能遇到的典型问题
- **相关菜单路径**：管理后台的菜单入口

**格式规范**：
- 使用标准 MD 语法（`## 标题`、`> 引用块`、MD 表格、代码块）
- `> 引用块` 渲染为提示框样式（info-box）
- 表格使用 MD 表格语法，自动应用斑马纹样式
- 代码块标注语言标识（如 ` ```json `）

### Step 6: 写入 MD 文件

写入 `admin_frontend/public/guide-docs/md/{file}.md`。

### Step 7: 更新 nav.json（新增章节时）

若为新增章节，在 `admin_frontend/public/guide-docs/nav.json` 中插入对应条目，`file` 字段与 MD 文件名（不含 `.md`）一致。

## 文件结构说明

```
admin_frontend/public/guide-docs/
├── index.html          ← Shell（无需修改）
├── nav.json            ← 导航配置（新增/删除章节时更新）
├── md/                 ← MD 源文件目录（维护对象）
│   ├── principles.md
│   ├── practices.md
│   ├── practices-memory.md
│   └── ...（共 22 个）
└── assets/             ← 静态库（无需修改）
```

### nav.json 结构

```json
[
  { "title": "显示名称", "file": "md文件名（不含.md）" },
  {
    "title": "带子菜单的分组",
    "file": "索引页文件名",
    "children": [
      { "title": "子章节", "file": "子章节文件名" }
    ]
  }
]
```

**约束**：
- `file` 值必须与 `md/` 目录下 MD 文件名（不含 `.md`）严格对应
- 仅支持一级子菜单（不支持三级嵌套）
- 新增/删除章节：同步修改 `nav.json` 和 `md/*.md`

### 五大块分类规则

| 章节 | 说明 |
|------|------|
| 基础原理 | 核心概念、架构理念、设计原则 |
| **最佳实践** | 按主题分篇（记忆策略等）；`practices.md` 为索引页，子页为 `practices-*.md` |
| 对外接口 | 接口列表、请求参数、响应格式、错误码 |
| 版本更新 | 版本号、更新日期、变更摘要、兼容性说明 |
| 操作说明 | 进入方式、字段说明表格、操作步骤、常见问题 |

---

## 关键约束

1. **MD 是最终产物**：直接写入 `guide-docs/md/`，浏览器自动渲染，无需 HTML 转换
2. **路径稳定**：禁止创建版本号子目录
3. **唯一入口**：`guide-docs/index.html`（无需修改）
4. **五大块**：基础原理、最佳实践、对外接口、版本更新、操作说明
5. **nav.json 同步**：新增或删除章节时必须同步更新 `nav.json`
