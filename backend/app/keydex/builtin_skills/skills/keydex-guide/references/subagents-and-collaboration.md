# Sub-Agent 与协作

## 适用场景

主 Agent 可以把边界清楚的子任务交给独立 Sub-Agent，并继续在同一任务中汇总结果。当前提供两种角色：

- **Explorer**：只读调研。适合搜索代码、阅读文件、追踪数据流、查资料并返回路径和证据；不能修改文件或运行写入型操作。
- **Worker**：执行型工作。适合在共享工作区内修改文件、运行命令和验证结果；权限仍受当前会话策略与审批约束。

两种角色继承主会话当前模型，但使用各自独立的子会话和上下文。Sub-Agent 不能再创建新的 Sub-Agent。

## 如何发起

直接告诉主 Agent 要委派的角色、目标、范围和交付物，例如“让 Explorer 只读梳理这条调用链，给出文件和行号”或“让 Worker 只改这两个文件并跑指定测试”。只有主 Agent 能创建新的 Sub-Agent。

委派后不需要用户手动轮询。主 Agent 会等待本次子任务进入完成、失败、取消或需要处理的状态，再取得最终报告。需要并行时，应把任务拆成互不写同一文件的独立范围。

## 在哪里查看

入口位于 **Agent 会话 → 右侧栏 → Sub-Agents**。列表按活动中、已完成、已取消和失败状态分组；会话时间线也会显示 Explorer 或 Worker 胶囊。点击条目或胶囊可打开子会话详情，查看完整对话、工具过程和结果。

在工作台中，还可把子会话作为助手的次级会话打开，并在相邻标签中查看它引用或修改的文件。Worker 产生的文件变更会并入主会话的文件变更历史，仍应由用户在统一审阅区确认。

## 中途引导与停止

当前 Sub-Agent 运行时，详情页提供“中途引导 Sub-Agent”输入框、发送引导和停止操作：

- 用中途引导补充范围、纠正方向或指定输出格式，不会新建另一个角色。
- 停止只终止当前运行，不会自动撤销已经写入的文件。
- 已完成、失败或历史运行是只读记录；要沿用同一角色和上下文继续工作，让主 Agent 继续该 Sub-Agent。

继续会复用原来的角色、历史和私有上下文，但创建一轮新的运行；不能把 Explorer 在继续时改成 Worker。

## 状态与边界

除运行中、完成、失败和取消外，Sub-Agent 还可能显示等待审批、等待用户输入或等待外部工具。此时打开详情处理具体请求，不要重复创建同样的子任务。

Sub-Agent 不是脱离当前任务的后台守护进程。它共享当前工作区，Worker 也可能与主 Agent 或其他 Worker 写入同一文件；委派前明确文件边界，完成后统一审阅实际变更。
