# 目标、计划与上下文整理

## 会话目标

目标用于让 Agent 在多轮任务中持续记住要达成的结果。一个会话同一时间只保留一个活动目标。

可以根据界面提供的入口：

- 创建目标。
- 查看目标内容和当前状态。
- 暂停或恢复。
- 编辑目标。
- 删除不再需要的目标。

暂停目标不会删除会话内容；恢复后应先查看此前进展。删除目标只移除目标约束，不等于撤销已经完成的工作。

## 什么时候设置目标

适合持续多轮、需要验收或容易偏离范围的工作，例如完整重构、逐项迁移、持续排查。一次性简单问答不必强制创建目标。

目标应描述最终结果和边界，不要把每个临时操作都写进目标。具体步骤可以交给本轮计划。

## 计划

计划用于展示较复杂任务的步骤和进度。用户可以根据计划判断当前在做什么、还剩什么，以及是否需要中途调整范围。

计划不是完成证明。验收时仍应查看最终内容、测试或页面结果。

## 上下文整理

会话很长时，可以使用 `/compact` 或界面提供的上下文整理入口。它用于保留关键结论并压缩较早内容，让后续对话继续进行。

整理后的上下文由三部分共同组成：

- 较早历史的逐轮工作账本。每个真实用户轮次分别记录“用户说了什么、Agent 做了什么、结果与错误、验证和结束状态”；长时间没有新用户消息的单轮任务，则追加独立执行片段，不把几个轮次合并成宽泛的阶段摘要。
- 被完整保留的近期轮次原文，包括该轮用户输入、Agent 回复和工具调用结果。文字、引用、评论、显式 Skill 和附件属于同一个结构化输入组；轮次被完整保留时，其完整结构化组也一起恢复，不会只留下用户原文或从零散历史里拼接半组。
- 最近的原始执行现场，例如最新一段回复和工具调用、当前计划，以及预算允许时的最近读取文件清单或小片段。

这些内容共用一份压缩后上下文预算，常态目标约为 2 万 tokens，只有不可拆的必要内容才使用最高约 2.4 万 tokens 的正常软余量。它不是给摘要、引用、Skill 和文件各分一份额度，因此整理后通常不会把各项上限简单相加。这份预算只计算压缩替换出来的历史；每次请求都会重新生成的系统说明和工具定义会另行计入最终请求，并继续接受当前模型的硬窗口检查。

Keydex 先按真实用户消息确定轮次边界。预算允许时，近期轮次以“用户消息 + Agent 回复 + 工具交换”的完整原文原子保留；更早轮次进入逐轮账本。若一个长任务只有最初一条用户指令但 Agent 已连续执行数小时，则在安全的消息单元边界切出较早执行片段进行压缩，并保留最近执行现场，不会因为“只有一轮”而把整段原文永久保留。

显式选择的 Skill 不会把旧的 Skill 正文直接复制进新上下文。Keydex 会保留激活信息，并在整理后的第一次有效执行中重新调用当前版本的 `load_skill`；这样仍会应用当前项目范围、优先级和权限。附件也会按当前会话、用户、文件存在性、类型和大小重新校验。

上下文整理不会无条件重新读取所有旧文件，也不会要求 Agent 读取一份隐藏的完整原对话文件。它可以保留最近读取文件的清单，并在共享预算有余量时附带当前文件片段；清单会跨连续压缩继承并用当前文件内容重新生成可选片段。后续是否再次调用文件工具，仍由 Agent 根据任务需要和当前权限决定。

目标负责描述持续任务的最终结果和生命周期，但不再单独复制一份创建目标时的引用、Skill 或附件。结构化恢复属于通用的上下文整理流程，因此普通会话、目标续跑和进行中的长任务遵守同一套规则。

执行前建议：

- 确认重要限制、未完成项和文件位置已经明确。
- 对极其关键的原文先单独保存或导出。
- 整理后检查目标、当前任务和下一步是否仍准确。

上下文整理不是删除会话，也不等于归档。
