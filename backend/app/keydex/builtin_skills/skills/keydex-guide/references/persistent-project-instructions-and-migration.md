# `keydex.md` 持久说明与迁移

## 它解决什么问题

用 `keydex.md` 保存每次新请求都应知道、但不值得在每轮重复说明的长期背景，例如项目边界、目录职责、常用命令、代码约定、验证要求和禁止事项。

不要把它当作 Skill、权限配置或秘密存储：它没有选择器和独立编辑页，不会创建工具、放宽审批，也不应包含密钥、令牌或个人隐私。多步骤工作流、配套脚本和按场景触发的专门知识应放进 Skill。

## 两个位置和生效范围

| 层级 | Windows 位置 | 生效范围 |
|---|---|---|
| 系统级 | `%USERPROFILE%\.keydex\keydex.md` | 普通 Chat 和所有项目会话 |
| 项目级 | `<项目目录>\.keydex\keydex.md` | 仅绑定该项目的会话 |

和常见 Agent 说明文件的层级对应如下：

| 用途 | Keydex | Codex | Claude Code |
|---|---|---|---|
| 当前用户的全局偏好 | `%USERPROFILE%\.keydex\keydex.md` | `~/.codex/AGENTS.md` | `~/.claude/CLAUDE.md` |
| 仓库共享说明 | `<项目>\.keydex\keydex.md` | 项目根及目录链中的 `AGENTS.md` | `<项目>/CLAUDE.md` 或 `<项目>/.claude/CLAUDE.md` |
| 更近目录的覆盖 | 当前无独立层 | 更近目录的 `AGENTS.md` 或 `AGENTS.override.md` | 子目录说明按需加载 |
| 仅本机的项目说明 | 当前无独立层 | 由仓库约定或备用文件名处理 | `<项目>/CLAUDE.local.md` |
| 导入其他说明 | 当前不支持 | 通过层级发现和备用文件名组织 | `@path` 导入 |

当前没有单独的 `keydex.md` 管理页面。请直接在上述位置创建 UTF-8 Markdown 文件；也可以让 Agent 创建，但必须先确保该路径处于当前文件访问范围。

项目会话同时读取两份文档，先应用系统级通用说明，再应用更具体的项目级说明。普通 Chat 只读取系统级说明。缺失、空白、超过 32 KiB、不是普通文件、包含无效文本，或经过符号链接/目录联接的文件会被跳过并产生诊断；一层失败不会阻止另一层正常生效。

修改后从下一次新请求开始生效，正在运行的一轮不会中途换规则。内容只在每次请求时临时加入模型上下文，不会变成一条聊天消息，也不会写入会话历史。

## 指令层级

按以下顺序解释冲突，越靠前优先级越高：

1. Keydex 系统提示词和安全约束。
2. 用户当前消息中的明确要求。
3. 当前项目的 `.keydex\keydex.md`。
4. 系统级 `.keydex\keydex.md`。
5. 默认行为。

因此，项目说明可以细化系统级偏好，但不能覆盖用户本轮明确要求或安全规则。文档也不能扩大文件、命令或 MCP 权限；这些能力应在“设置 → 策略配置”及对应的 MCP 设置中配置。

## 推荐内容结构

保持短小、明确、可执行，优先写模型无法从代码立即判断的信息：

```markdown
# 项目工作说明

## 项目与范围
- 本仓库负责什么；哪些目录或系统不在本次范围内。

## 目录地图
- `backend/`：服务端。
- `desktop/`：桌面端。

## 环境与命令
- Python 使用项目虚拟环境。
- 只运行与改动范围相关的测试；发布前再跑完整构建。

## 工程约定
- 保留用户已有改动，不处理无关文件。
- 修改前先确认真实运行路径和数据边界。

## 验证与完成标准
- UI 改动验证可见结果；后端改动覆盖成功和失败路径。
- 交付时列出改动文件、验证结果和已知边界。

## 风险与禁止事项
- 不提交密钥，不绕过审批，不执行破坏性清理。
```

## 从 `AGENTS.md` 迁移

Codex 通常支持用户级 `~/.codex/AGENTS.md`，并从项目根到当前目录逐层发现 `AGENTS.md` 或覆盖文件。迁移到 Keydex 时：

- 把真正跨项目的用户偏好整理进系统级 `keydex.md`。
- 把仓库级规则整理进项目级 `keydex.md`。
- Keydex 当前不递归加载子目录说明，也没有 `AGENTS.override.md` 和备用文件名；把各层仍然必要的规则合并为一份项目说明，并删除相互冲突或只对旧目录有效的内容。
- 与某类任务绑定的详细流程不要全部塞进项目说明，应拆成项目级 Skill，并在其 `SKILL.md` 中写清触发条件。

## 从 `CLAUDE.md` 迁移

Claude Code 常见层级包括用户级 `~/.claude/CLAUDE.md`、项目级 `CLAUDE.md` 或 `.claude/CLAUDE.md`，以及仅本机使用的 `CLAUDE.local.md`，还支持 `@path` 导入。迁移到 Keydex 时：

- 用户级内容对应系统级 `keydex.md`；项目共享内容对应项目级 `keydex.md`。
- `CLAUDE.local.md` 没有一一对应的项目私有层。真正跨项目的本机偏好可放系统级文件；只对一次任务有效的内容留在会话中。不要为了模拟本机层而把秘密写进项目文件。
- Keydex 当前不支持 `@` 导入或按子目录延迟加载。只合并必要正文；较长参考资料和工作流改为 Skill 的 `references/` 与 `scripts/`。
- Claude Code 中的权限、Hook、工具白名单或配置项不能迁进说明文本，应在 Keydex 的策略、MCP 或相关设置页重新配置。

迁移后用一个新会话验证三件事：普通 Chat 只看到系统说明；项目会话同时遵循通用和项目说明；本轮明确要求与说明冲突时，以本轮要求为准。

原工具的层级或文件名可能继续变化。需要核对迁移源时查看 [Codex 的 AGENTS.md 官方说明](https://learn.chatgpt.com/docs/agent-configuration/agents-md) 和 [Claude Code 的 CLAUDE.md 官方说明](https://code.claude.com/docs/en/memory)；Keydex 的目标位置与优先级仍以本页和当前应用版本为准。
